## 引言

当我们谈论 AI Agent 时，往往会把所有注意力放在"模型有多强"上。但 Anthropic 的工程师团队在和数十个团队合作后发现：**最成功的 Agent 实现，不是靠更复杂的模型，而是靠更清晰的结构**。

本文源自 Anthropic 官方工程文档，结合实践验证的系统方法论，帮助你理解：

- Agent 的本质是什么
- Harness（马具）在 Agent 系统中扮演什么角色
- 如何从简单到复杂地构建 Agent 系统
- 如何设计出可靠、可控、可信赖的 Agent

---

## 第一部分：基础概念

### 什么是 Agent？

传统 chatbot 是问答机器：用户问，模型答，一问一答，独立完成。

**Agent 则不同**——它是**自驱动的循环系统**：

```
用户给出目标 → Agent 规划步骤 → 执行动作 → 观察结果 → 调整计划 → 重复直到完成
```

以报销差旅发票为例，Agent 会：

1. **规划**：先转录照片，再提取金额和商户，分类 expense，最后提交到公司系统
2. **执行**：一步步来做
3. **观察**：如果酒店住宿费超过每日上限被标记，Agent 会思考"是否需要先查一下公司的报销政策？"
4. **调整**：在继续前，主动停下来向用户确认

```
┌─────────────────────────────────────────────────────────────────┐
│  Agent 的核心特征：                                              │
│  • 动态规划：不是预设脚本，而是自己决定怎么做                     │
│  • 工具使用：能调用外部服务完成任务                              │
│  • 环境反馈：根据执行结果调整下一步行动                           │
│  • 人类介入：在关键节点停下来等待确认                             │
└─────────────────────────────────────────────────────────────────┘
```

### 两个关键区分：Workflows 与 Agents

Anthropic 将所有 Agent 相关系统统称为 **agentic systems**，但区分了两大类：

| 类型 | 定义 | 适用场景 |
|------|------|----------|
| **Workflows** | 通过**预定义代码路径**编排 LLM 和工具 | 任务明确、流程固定 |
| **Agents** | LLM **动态主导**自己的流程和工具使用 | 需要灵活性、模型驱动决策 |

**选择原则**：能用简单方案就不要引入 Agent。Agent 带来的延迟和成本换取的是更好的任务完成度，问问自己这个交换值不值。

## 第二部分：核心框架——Agent 的四层组件

Anthropic 将 Agent 系统分解为**四层组件**，每一层既是能力的来源，也是潜在的监督漏洞：

```
┌─────────────────────────────────────────────────────────────────┐
│                      Agent 系统四层组件                           │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│    ┌─────────────────────────────────────────────────────────┐   │
│    │                    The Model                            │   │
│    │              （模型：核心智能）                          │   │
│    │  训练过程塑造了模型知道什么、如何推理、如何行为          │   │
│    └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│    ┌─────────────────────────────────────────────────────────┐   │
│    │                     The Harness                          │   │
│    │           （Harness：指令与护栏）                        │   │
│    │  告诉 Agent 什么该做、什么不该做、在哪里停下来          │   │
│    └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│    ┌─────────────────────────────────────────────────────────┐   │
│    │                      The Tools                           │   │
│    │              （工具：外部能力扩展）                       │   │
│    │  没有工具，Agent 只能读不能做                            │   │
│    └─────────────────────────────────────────────────────────┘   │
│                              │                                   │
│    ┌─────────────────────────────────────────────────────────┐   │
│    │                   The Environment                        │   │
│    │              （环境：运行上下文）                        │   │
│    │  同一 Agent 在不同环境下有不同的数据访问和风险级别       │   │
│    └─────────────────────────────────────────────────────────┘   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 2.1 The Model（模型层）

模型是"智能"的来源。这种智能来自训练过程，决定了模型知道什么、如何推理、如何行为。

**模型层的关键点**：
- 模型能力直接决定了 Agent 能做什么
- 新模型发布可能显著改变 Agent 的能力边界
- 但再强的模型，如果其他层配置不当，也会出问题

### 2.2 The Harness（马具层）

这是 **Harness Engineering** 名字的来源。

**Harness 的定义**：模型运行时所遵循的**指令和护栏**。

类比：驯马时给马装上马具，不是限制马的能力，而是让马能按骑手的意图工作。

**Harness 的作用**：
- 告诉 Agent 哪些动作需要用户确认
- 设置行为边界（如：超过 100 元的支出要标记）
- 定义何时应该停下来等待人工介入

**Harness 的重要性**：
> 一个训练良好的模型，仍可能因为 Harness 配置不当、工具过于宽松、或环境暴露过多而被 exploitation。

### 2.3 The Tools（工具层）

工具是 Agent 与外部服务/应用交互的接口。

**没有工具的 Agent**：
- 能读报销收据的照片 ✓
- 但不能把它提交到报销系统 ✗

**工具设计原则**（Anthropic 称之为 ACI——Agent-Computer Interface）：

1. **让格式贴近模型在互联网上自然见过的形式**
2. **避免格式开销**：不要让模型去数代码行数、做字符串转义
3. **给足够的 tokens 让模型"思考"再行动**
4. **像写给 junior developer 的 docstring 一样写工具描述**

### 2.4 The Environment（环境层）

环境是 Agent 运行的实际场所。

**环境决定**：
- Agent 能访问哪些文件、网站、系统
- 同一 Agent 在公司内网 vs 个人手机上，风险级别完全不同

**环境配置要点**：
- 沙箱隔离生产环境
- 明确的权限边界
- 审计日志记录所有操作

## 第三部分：构建模式——从简单到复杂

构建 Agent 系统时，Anthropic 推荐**渐进式复杂度**策略：

```
简单增强 LLM
      ↓
┌─────────────────────────────────────┐
│         Workflows（工作流）           │
│  ┌─────────┐ ┌─────────┐ ┌───────┐  │
│  │Prompt   │ │Routing  │ │Parallel│  │
│  │Chaining │ │         │ │ization │  │
│  └─────────┘ └─────────┘ └───────┘  │
│  ┌───────────────┐ ┌─────────────┐  │
│  │Orchestrator-  │ │Evaluator-   │  │
│  │workers        │ │Optimizer    │  │
│  └───────────────┘ └─────────────┘  │
└─────────────────────────────────────┘
      ↓
┌─────────────────────────────────────┐
│         Agents（智能体）              │
│  自主循环：计划→行动→观察→调整        │
└─────────────────────────────────────┘
```

### 3.1 基础模块：增强型 LLM（Augmented LLM）

所有 Agent 系统的基础构建块是**增强型 LLM**——不只是原始的文本生成器，而是配备了：

- **检索（Retrieval）**：能从知识库获取信息
- **工具（Tools）**：能调用外部服务
- **记忆（Memory）**：能保存和复用信息

**实现建议**：使用 **Model Context Protocol (MCP)** 来标准化工具集成，它提供了简单的客户端实现，能接入不断增长的第三方工具生态。

### 3.2 Workflow 1：Prompt Chaining（提示链）

**模式**：将任务分解为**顺序步骤**，每个 LLM 调用处理前一个的输出。

```
Task → LLM 1 → LLM 2 → LLM 3 → ... → Output
         ↓        ↓
       Gate     Gate  (可选的检查点)
```

**何时使用**：任务能干净地分解成固定子任务，主要目标是**用延迟换准确度**——每个 LLM 调用只做一件更容易的事。

**典型场景**：
- 生成营销文案 → 翻译成另一种语言
- 写大纲 → 检查大纲是否符合标准 → 根据大纲写正文

### 3.3 Workflow 2：Routing（路由）

**模式**：对输入分类，导向不同的专门处理路径。

```
Input → Classifier → [专门处理路径 A]
                   → [专门处理路径 B]
                   → [专门处理路径 C]
```

**何时使用**：复杂任务有明确的不同类别，每类需要分开处理且分类能准确完成。

**典型场景**：
- 客户服务：普通问题 / 退款请求 / 技术支持 → 不同流程
- 模型路由：小问题用小模型（省钱），复杂问题用大模型（保质量）

### 3.4 Workflow 3：Parallelization（并行化）

LLM 能同时处理任务并通过程序聚合输出，表现为两种形式：

**Sectioning（分段）**：将任务分解为独立子任务并行执行
**Voting（投票）**：用相同任务多次执行获得不同输出

```
┌─────────────────────────────────────────────────┐
│  Parallelization                                  │
│                                                   │
│  Input                                            │
│    │                                              │
│    ├──→ Subtask A ──┐                            │
│    ├──→ Subtask B ──┼──→ Aggregator → Output     │
│    └──→ Subtask C ──┘                            │
│                                                   │
│  或                                              │
│                                                   │
│  Input ──→ LLM 1 ──┐                             │
│         ──→ LLM 2 ──┼──→ Voting → Output         │
│         ──→ LLM 3 ──┘                             │
└─────────────────────────────────────────────────┘
```

**何时使用**：子任务可以并行以提升速度，或需要多视角/多次尝试以获得更高置信度。

**典型场景**：
- **Sectioning**：一个模型处理用户查询，另一个筛选不当内容/请求（比同一个 LLM 同时处理两项效果更好）
- **Voting**：多个 prompt 审查代码漏洞，少数服从多数

### 3.5 Workflow 4：Orchestrator-Workers（编排器-工作者）

**模式**：中央 LLM 动态分解任务，委托给工作者 LLM，最后综合结果。

```
                    ┌───────────────────────┐
                    │      Orchestrator     │
                    │   （中央 LLM 动态分解）│
                    └───────────┬───────────┘
                                │
            ┌───────────────────┼───────────────────┐
            ▼                   ▼                   ▼
      ┌───────────┐       ┌───────────┐       ┌───────────┐
      │  Worker 1 │       │  Worker 2 │       │  Worker 3 │
      │  (子任务) │       │  (子任务) │       │  (子任务) │
      └─────┬─────┘       └─────┬─────┘       └─────┬─────┘
            └───────────────────┴───────────────────┘
                                │
                    ┌───────────┴───────────┐
                    │    Synthesizer       │
                    │    （综合结果）       │
                    └─────────────────────┘
```

**与 Parallelization 的关键区别**：子任务不是预定义的，而是由编排器根据具体输入动态决定的。

**何时使用**：复杂任务，无法提前预测需要哪些子任务（如代码修改的文件数量和性质取决于具体任务）。

**典型场景**：
- 每次修改多个文件的代码产品
- 从多个来源搜集和分析信息的搜索任务

### 3.6 Workflow 5：Evaluator-Optimizer（评估器-优化器）

**模式**：一个 LLM 生成响应，另一个提供评估和反馈，循环迭代。

```
Response ──→ Evaluator ──→ Feedback ──→ LLM ──→ Response
                                  ↑                      │
                                  └──────────────────────┘
                                    (迭代改进)
```

**何时使用**：有明确的评估标准，迭代改进能带来可衡量的价值。

**典型场景**：
- 文学翻译：翻译 LLM 捕捉不到的细微之处，评估 LLM 提供有用批评
- 复杂搜索：评估 LLM 决定是否需要进一步搜索

### 3.7 Agents（智能体）

当以上 Workflow 都不够用时——任务无法预测步骤数量、无法硬编码固定路径——就需要真正的 Agent。

**Agent 的工作方式**：

```
┌─────────────────────────────────────────────────────────────────┐
│                      Autonomous Agent Loop                        │
│                                                                  │
│    1. 接收任务（来自用户命令或交互讨论）                          │
│    2. 规划步骤                                                   │
│    3. 独立操作，可能返回向人类获取更多信息                        │
│    4. 从环境获取"ground truth"（工具结果、代码执行）             │
│    5. 在检查点或遇到 blockers 时暂停等待反馈                      │
│    6. 达到完成条件或最大迭代次数时停止                           │
└─────────────────────────────────────────────────────────────────┘
```

**何时使用 Agent**：
- 开放式问题，难以或不可能预测所需步骤
- 需要模型驱动决策的灵活性
- LLM 需要运行多轮
- 必须在受信任环境中

**Agent 的实现往往比想象的简单**：它通常就是 LLM + 工具 + 基于环境反馈的循环。**关键是工具集和工具文档的清晰设计**。

## 第四部分：核心原则

Anthropic 在构建 Agent 时遵循三个核心原则：

### 原则一：保持简单（Maintain Simplicity）

```
复杂度的代价：
• 延迟增加
• 成本上升
• 错误累积
• 可维护性下降

建议：
✓ 先用简单 prompt 试试
✓ 用 evaluation 优化
✓ 只有在简单方案不够用时才加复杂度
```

### 原则二：优先透明（Prioritize Transparency）

Agent 应该能展示自己的规划步骤，让用户理解它在做什么、为什么这样做。

**实现方式**：
- 在关键决策点停下来解释
- 显示 Agent 的推理过程
- 记录和审计所有重要行动

### 原则三：精心设计 ACI（Agent-Computer Interface）

ACI = Agent-Computer Interface，类比 HCI（Human-Computer Interface）

**核心思想**：**花多少心思在人机交互上，就花多少心思在 Agent-计算机交互上**。

**ACI 设计检查清单**：

| 问题 | 你应该问自己 |
|------|-------------|
| 工具名称 | 从描述和参数就能知道怎么用吗？还是要仔细思考？ |
| 参数名 | 够直观吗？像给 junior developer 写 docstring |
| 错误处理 | 人类程序员会怎么犯错？工具能否 poka-yoke（防呆）？ |
| 示例 | 有没有 example usage、edge cases、输入格式要求？ |
| 边界 | 工具和其他工具的边界清晰吗？ |

**实践案例**：Anthropic 在构建 SWE-bench Agent 时发现，模型在 agent 离开根目录后使用相对文件路径会出错。解决方案：改为**强制使用绝对路径**——模型此后完美使用，再没出错。

## 第五部分：工具设计最佳实践

### 格式选择的三个建议

1. **给模型足够的 tokens "思考"**，再让它写，避免把自己逼进死角
2. **格式贴近模型在互联网上自然见过的形式**
3. **避免格式开销**：不要让模型去数代码行数、做字符串转义

### 反面例子 vs 正面例子

```
❌ 反面：Diff 格式
--- file.js
+++ file.js
@@ -1,3 +1,4 @@
+ new line

问题：需要在知道新代码写什么之前就知道改几行

✅ 正面：直接重写整个文件
或者用更清晰的格式
```

### Poka-yoke（防呆）原则

改变参数形式，让错误更难发生。

**例子**：不用相对路径，用绝对路径。

## 第六部分：框架使用指南

### 常见框架

| 框架 | 特点 |
|------|------|
| Claude Agent SDK | 官方，简洁 |
| Strands Agents SDK (AWS) | AWS 集成 |
| Rivet | 拖拽式 GUI |
| Vellum | GUI 工具 |

### 框架的利弊

**优点**：
- 快速起步
- 简化标准任务（LLM 调用、工具定义、链式调用）

**缺点**：
- 额外抽象层可能掩盖底层 prompt 和响应
- 增加复杂度
- 容易在简单方案够用时也引入框架

### Anthropic 的建议

**先用 LLM API 直接构建**——很多模式只需几行代码就能实现。

如果使用框架，**确保你理解底层代码**。对底层假设的错误是常见的错误来源。

## 第七部分：实际案例

### 案例一：客户支持 Agent

**为什么适合 Agent**：
- 支持交互天然是对话流程，需要外部信息和操作
- 工具可以集成获取客户数据、订单历史、知识库文章
- 退款、更新工单等操作可以程序化处理
- 成功与否可以通过用户定义的解决方案清晰衡量

**验证**：几家公司已经通过**按成功解决收费**的模式验证了可行性。

### 案例二：编码 Agent

**为什么适合 Agent**：
- 代码解决方案可以通过自动化测试验证
- Agent 可以用测试结果作为反馈迭代
- 问题空间定义良好、结构化
- 输出质量可以客观衡量

**SWE-bench 成就**：Anthropic 的 Agent 现在可以仅根据 PR 描述解决真实的 GitHub issues。但自动化测试验证功能，人类审查确保解决方案符合更广泛的系统要求。

## 第八部分：安全与信任

### 五项核心原则（Anthropic）

1. **人类控制**：保持人类在控制位
2. **人类价值对齐**：与人类价值观对齐
3. **交互安全**：保护 Agent 交互的安全
4. **透明性**：保持透明
5. **隐私**：保护隐私

### 关键风险

**Agent 面临的独特风险**：

1. **Prompt Injection**：攻击者试图通过输入欺骗模型执行不该做的动作
2. **权限过大**：Agent 拥有的权限超出必要
3. **缺乏监督**：自主性过高导致无法及时发现和纠正错误

### 安全建议

```
安全检查清单：
□ 最小权限原则：Agent 只应该拥有完成任务的最小权限集
□ 沙箱隔离：在隔离环境中测试后再上生产
□ 检查点机制：在关键操作前停下来等待确认
□ 审计日志：记录所有重要操作以便追溯
□ 失败策略：定义清晰的失败降级路径
□ 停止条件：设置最大迭代次数防止无限循环
```

```
│                  → Agents                                        │
│                                                                  │
│  3. 重视 Harness：它告诉模型什么该做、什么不该做、               │
│                   在哪里停下来                                  │
│                                                                  │
│  4. 精心设计工具：ACI 值得你花和 HCI 一样多的心思                 │
│                                                                  │
│  5. 保持透明：让用户理解 Agent 在做什么                           │
│                                                                  │
│  6. 安全第一：最小权限、沙箱、审计、停止条件                      │
│                                                                  │
│  7. 度量和迭代：测量性能，持续改进                               │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

> **核心理念**：LLM 领域的成功不在于构建最复杂的系统，而在于构建**最适合你需求的系统**。用简单的 prompt 起步，用全面的评估优化，只有在简单方案不够用时才引入多步骤 Agent 系统。
